# Architecture Decisions — Phase 1

Generated by /plan-eng-review on 2026-04-13
Branch: main

## Context

Phase 1 validates whether Gemma4-26B via llama.cpp produces useful BTC/USDT trading signals.
This is the "C" in the approved "C then B" approach from the design doc (`docs/design.md`).
Build the decision harness + historical replay before touching Freqtrade or live trading.

## Decisions

| # | Decision | Choice | Why |
|---|----------|--------|-----|
| 1 | Candle timeframe | 15-minute | Balances signal resolution vs replay runtime. 2,880 candles for 30 days. 1h was too coarse, 1m explodes compute (43K LLM calls). |
| 2 | LLM JSON output | Grammar-constrained via `extra_body={"json_schema": ...}` | llama.cpp supports this on /v1/chat/completions. Prevents malformed JSON at token level instead of catching after the fact. Keeps replay data clean. |
| 3 | Project structure | Python package `llm_local/` + `tests/` | Near-zero cost now, avoids reorganization when Phase 2 adds Freqtrade. |
| 4 | Prompt context | Full TA indicators (RSI, MACD, BB, SMA) via pandas-ta + raw OHLCV | Gives the model the richest context. pandas-ta is pure Python, no C deps. |
| 5 | Configuration | TOML file + env vars for secrets | `tomllib` is stdlib in Python 3.12+. Secrets from env (BINANCE_API_KEY, etc). More flexible than dataclass-only. |
| 6 | Schema | SQLite: candles, decisions, replay_runs. No outcomes table. | Outcomes computed at analysis time by joining decisions to candles. Keeps decisions as clean immutable records. Outcomes table deferred to Phase 2 (real trades have non-deterministic fills). |
| 7 | Harness design | Pure function, no side effects | Caller validates action legality (can't sell with no position). Makes harness testable in isolation and reusable across all phases. |
| 8 | Prompts | Python module `prompts.py` with named template functions | Version-controlled, IDE-friendly. Each replay run records which prompt version was used. |
| 9 | Exit logic | LLM decides SELL + 24-candle (6h) max hold cap | Model gets to exercise exit timing (part of the signal we test), but max hold prevents "buy and forget" positions that distort metrics. |
| 10 | Replay performance | Checkpoint/resume + progress bar + quick mode (200 candles) | Full replay takes 8-12 hours. Checkpoint prevents losing hours on crash. Quick mode enables fast prompt iteration (~30 min). |
| 11 | Data recency | Most recent 30 days (past Gemma4 training cutoff) | Outside voice flagged that old data measures memorization, not prediction. Recent data (2026) is past the model's training window. |
| 12 | Tests | Full pytest suite, 92 tests, written alongside each module | User preference: "well-tested code is non-negotiable." AI-assisted test writing makes full coverage near-free. |

## Trading Strategy

There is no fixed trading strategy in Phase 1. The LLM receives market data (candles + indicators + position state) and makes buy/sell/hold decisions based on the prompt template (`llm_local/prompts.py`). The prompt gives guidelines ("consider trend, momentum, volatility, volume") but doesn't prescribe a specific approach (mean reversion, momentum, breakout, etc.).

Phase 1's purpose is to determine whether the LLM produces any useful signal at all. If it does, Phase 3's meta-loop (Claude Opus) analyzes what patterns work and refines the prompt to sharpen them.

```
     |
     v

     |  fetch_ohlcv() -> Binance API (paginated, cached in SQLite)
     |  compute_indicators() -> pandas-ta (RSI, MACD, BB, SMA)
     v
SQLite [candles table]
     |
     v
replay.py
     |  for each candle (skip first 34 for indicator warmup):
     |    build context window (last 24 candles + indicators + position)
     |    --> harness.py
     |         |  prompts.py builds prompt from template
     |         |  call llama.cpp /v1/chat/completions
     |         |    extra_body={"json_schema": Decision schema}
     |         |  parse Decision (action, size_pct, confidence, reasoning)
     |         v
     |    validate action legality (can't sell with no position)
     |    update simulated position
     |    enforce 24-candle max hold cap
     |    write Decision to SQLite [decisions table] immediately
     |    log: candle_idx, action, confidence, latency_ms
     |    checkpoint: resume from last decision on restart
     v
SQLite [decisions + replay_runs tables]
     |
     v
analysis.py
     |  join decisions to candles -> compute outcomes
     |  metrics: win rate, avg P&L, Sharpe ratio, max drawdown
     |  confidence calibration: Pearson r(confidence, realized_return)
     |  compare across prompt versions
     v
stdout report
```

## Storage Architecture

All trading operations and agent reasoning are stored in **SQLite** via SQLAlchemy:

### candles table
- Cached OHLCV data from Binance (never re-fetched unless --force)
- Pre-computed technical indicators: SMA(24), SMA(96), RSI(14), MACD(12,26,9), Bollinger Bands(20,2)
- Unique constraint on (symbol, timeframe, timestamp_ms)
- First ~34 candles have NULL indicators (warmup period)

### decisions table
- Every LLM decision: action, size_pct, confidence, reasoning text
- `raw_response`: the complete verbatim LLM JSON output (for debugging prompt quality)
- `latency_ms`: inference time per decision
- `is_fallback`: 1 if LLM failed and defaulted to hold
- Foreign keys to replay_runs and candles

### replay_runs table
- Metadata per run: prompt_version, full config JSON, status, timestamps
- Enables comparing run A (prompt v1) vs run B (prompt v2)
- `quick_mode` flag tracks whether this was a 200-candle quick run

### MemPalace (Phase 3+, not yet wired)
- ChromaDB + SQLite knowledge graph
- Will give Claude Opus cross-session memory for strategy reviews
- Dependency installed but unused in Phase 1

## Operational Learnings

Discovered during /office-hours and /plan-eng-review sessions:

1. **Freqtrade sync blocker** (confidence 9/10): Freqtrade's strategy API is synchronous. LLM inference takes 3-30s. Integration MUST use pre-computed signal files, not inline calls. Otherwise backtester deadlocks.

2. **Validate signals first** (confidence 8/10): 95% of AI trading bots fail within 90 days because the signals are noise. Prove signal quality before building plumbing.

3. **llama.cpp JSON schema on OpenAI-compat API** (confidence 9/10): Grammar-constrained JSON works via `extra_body={"json_schema": schema}` on `/v1/chat/completions`. No need for native API.

4. **Gemma4-26B inference speed** (confidence 7/10): MoE model (4B active params) at Q4_K_XL with full TA context: expect 10-15s per call. A 2,880-candle replay takes 8-12 hours, not 4.

5. **TA indicator warmup NaN** (confidence 9/10): MACD needs 34 candles, RSI needs 14, BB needs 20. First 34 candles have NaN indicators. Skip in analysis to avoid corrupting metrics.

## Outside Voice Findings (from independent Claude subagent review)

The outside voice raised 9 points. Three were accepted and incorporated:

1. **Exit logic unspecified** -> Added 24-candle max hold cap + LLM-decided SELL (Decision #9)
2. **Data memorization risk** -> Using most recent 30 days past training cutoff (Decision #11)
3. **Replay time underestimated** -> Added quick mode for iteration, adjusted expectations to 8-12h (Decision #10)

Six were evaluated and resolved:
- SQLAlchemy overcomplexity: Already a dep, needed for Phase 2+
- Freqtrade integration backwards: Contradicts validated "C then B" approach
- 38 tests scope inversion: User preference for full coverage
- Grammar constraints via OpenAI client impossible: Verified it works
- TA indicator warmup: Already in test plan, handled by skipping first 34 candles
- Confidence calibration unmeasurable: Phase 1 is designed to test this. If noise, that's a finding.

## Success Criteria

- LLM produces coherent JSON decisions (grammar constraints ensure structure)
- Win rate measurably better than random (>52%)
- Confidence calibration: Pearson r > 0.1 between confidence and realized return
- System runs end-to-end without crashes on quick mode

## Problem Statement

Build a fully autonomous crypto trading system that runs on local hardware, uses local LLMs (Gemma4-26B via llama.cpp) for fast execution decisions and Claude Opus for deep strategy research, and generates meaningful income without human intervention. The system should monitor markets, reason about positions, execute trades on Binance, and learn from outcomes... all while the builder sleeps.

## What Makes This Cool

Three things make this genuinely interesting:

1. **Local-first AI trading.** Most LLM trading experiments depend on cloud APIs, which means latency, cost, and rate limits. Running Gemma4-26B locally on your own hardware means zero-cost inference with full control. Every decision the system makes costs you nothing.

2. **Two-tier reasoning.** The local LLM handles fast, routine decisions (should I buy/sell/hold right now?) while Claude Opus handles slow, deep decisions (is this strategy still working? what should we try next?). This mirrors how real trading desks work... junior traders execute, senior strategists set direction.

3. **Self-monitoring meta-loop.** (From second opinion) The system doesn't just trade, it watches itself. It detects when its own edge is decaying (rolling Sharpe degradation, win-rate drift) and pages Claude Opus to diagnose and propose strategy adjustments before it blows up. This is the difference between a bot that works for a month and an adaptive agent that survives regime changes.

## Constraints

- Local hardware only for execution-speed decisions (llama.cpp server at localhost:8080)
- Claude Opus API budget is a real cost, so strategy calls should be batched (nightly, not per-trade)
- Binance as the exchange (testnet for debugging, live with small capital for real validation)
- Python ecosystem (already committed via uv + deps)
- Solo builder, learning-oriented... the journey matters as much as the destination

## Premises

1. A local LLM (Gemma4-26B) can make useful real-time trading decisions better than simple rule-based logic
2. Backtesting has limited predictive value (R^2 < 0.025 vs live) but is necessary for development and prompt iteration
3. Two-tier model split (local LLM for execution, Claude Opus for strategy) is the right architecture
4. Paper trading is a debugging tool only. Real validation requires live trading with small real capital. Progression: backtest -> paper trade (debug) -> live small stakes -> scale up
5. Full autonomy is the goal, but observability is what makes autonomy trustworthy

## Approaches Considered

### Approach A: Ground-Up Learning Build
Build every layer from scratch: exchange client, backtester, LLM harness, trade engine, monitoring. Maximum learning, XL effort (4-8 weeks). Risk: getting stuck on plumbing and never reaching the interesting part. Pro: understanding every line of code.

### Approach B: Freqtrade + LLM Layer
Use Freqtrade for battle-tested infrastructure. Build the LLM decision layer and meta-loop on top. Medium effort (1-2 weeks to first backtest, 3-4 weeks to live). Pro: skip 3-4 weeks of plumbing. Con: Freqtrade has opinions about strategy structure.

### Approach C: Decision Harness First (Lean Validation)
Build ONLY the LLM decision harness + historical replay engine. No exchange, no live trading, no framework. Small effort (one weekend). Answers the biggest unknown: does the LLM produce useful signals? Can plug into either A or B afterward.

## Recommended Approach

**C then B.** Start with the LLM decision harness (Approach C) to validate that Gemma4-26B produces coherent, useful trading signals. This is a weekend project. Once validated, graduate to Freqtrade infrastructure (Approach B) for production-grade order management, backtesting, and exchange integration. Layer the meta-loop (self-monitoring + Claude Opus strategy review) on top.

### Phase 1: Decision Harness (Weekend)
- `decision_harness.py`: function that takes market context (OHLCV candles, current position, recent trades), sends structured prompt to local llama-server, receives JSON decision: `{"action": "buy"|"sell"|"hold", "size_pct": float, "reasoning": str, "confidence": float}`. **LLM failure contract:** if the LLM returns malformed JSON, times out (>30s), or is unreachable, the harness defaults to `{"action": "hold", "confidence": 0.0}` and logs the failure. In replay mode, failures are counted in the analysis. In live mode, 3 consecutive failures trigger a circuit-breaker that pauses trading and sends an alert.
- `data_fetcher.py`: pull 30 days of BTC/USDT OHLCV data from Binance public API, save to disk
- `replay_engine.py`: iterate through historical data tick-by-tick, call decision harness at each candle, record decisions + subsequent price movements to SQLite
- `analysis.py`: read results, compute win rate, avg hold time, simulated P&L (0.1% fee per side), confidence calibration (does stated confidence correlate with outcome?)
- **Success criteria:** coherent decisions, non-random win rate, meaningful confidence calibration

### Phase 2: Freqtrade Integration (Week 1)
- Install Freqtrade, configure for Binance testnet
- **Integration approach:** pre-computed signal file. The LLM decision harness runs as a separate process, writes signals to a JSON/CSV file on disk. Freqtrade strategy class reads from this file instead of calling the LLM synchronously. This avoids the sync-interface + LLM-latency blocker (Freqtrade's strategy API is synchronous, LLM inference takes 3-30s per call, which would deadlock backtest loops and block live event loops).
- Port the decision harness prompt into the signal-generation process
- Run Freqtrade's built-in backtester against pre-computed signals as integration smoke test
- Run on testnet for live integration verification

### Phase 3: Meta-Loop + Claude Opus (Weeks 2-3)
- SQLite table tracking every trade with outcome metadata
- Nightly cron job (runs at 00:00 UTC): compute rolling 7-day Sharpe, win-rate over trailing 50 trades, max drawdown over trailing 7 days
- **Degradation thresholds (trigger Claude Opus review):**
  - Rolling 7-day Sharpe drops below 0.5 (or below 50% of the 30-day baseline)
  - Win rate drops more than 10 percentage points from 30-day baseline (only active after 20+ trades recorded in trailing window)
  - Max drawdown exceeds 8% over any 7-day window
- When any threshold is breached: call Claude Opus with last 30 days of performance data
- Claude returns structured JSON strategy adjustment
- **Rollback mechanism:** before applying any strategy adjustment, snapshot the current parameters to a `strategy_history` table. If the adjusted strategy's 48-hour rolling Sharpe is more than 0.3 below the prior strategy's trailing 30-day Sharpe, auto-revert to the previous parameters and log the failed adjustment for Claude's next review. **Sparse-data guard:** if fewer than 10 trades occur in the 48-hour window, extend the evaluation window by 24 hours (up to 96h max) before reverting.
- Local LLM picks up adjusted parameters on next cycle
- MemPalace integration for cross-session context persistence

### Phase 4: Live Trading (Week 4+)
- Small real capital on Binance (start with $50-100)
- **Hard-coded risk rules (not LLM-decided):**
  - Max position size: 25% of portfolio per trade
  - Max daily drawdown: 5% of portfolio, system pauses and alerts if breached
  - Max total drawdown: 15% of portfolio, system halts completely
  - Single trade stop-loss: 2% of portfolio value
- Observability: logging every decision with reasoning, P&L tracking, alerts on anomalies
- Approval gates for early live period (optional human override above certain trade sizes)
- Gradual scale-up as confidence builds

## Open Questions

1. **Which crypto pairs?** BTC/USDT only to start, or multiple pairs from day one?
2. **Timeframe?** 1-minute candles (scalping) vs 1-hour (swing) vs daily? This affects everything about the LLM prompt design, latency requirements, and how quickly meta-loop baselines become valid (50 trades at 1h+ timeframes may take weeks to accumulate with small capital).
3. **Prompt engineering:** What context window does the LLM need? Last 24 candles? Technical indicators pre-computed? Raw OHLCV only? Phase 1 replay will answer this empirically.
4. **Claude Opus budget:** How many strategy review calls per month is acceptable? At ~$15/M input tokens + $75/M output, nightly calls with 30 days of trade data could add up. Estimate: ~$5-10/month if calls are triggered only on degradation thresholds rather than every night.

## Success Criteria

- **Phase 1:** LLM produces coherent JSON decisions. Win rate is measurably better than random (>52%). Confidence calibration: Pearson correlation between the `confidence` field in the JSON response and the realized return (price change over next N candles, where N matches the average hold time) must exceed 0.1.
- **Phase 2:** Freqtrade integration smoke test passes: strategy class calls LLM correctly, orders execute on testnet, trade data records to SQLite. System runs on testnet without crashes for 48+ hours. (Note: backtest results here validate system integration, not strategy performance, per Premise #2.)
- **Phase 3:** Meta-loop detects a simulated performance degradation and triggers a strategy adjustment. Claude Opus produces actionable strategy modifications.
- **Phase 4:** Live trading with small capital. System runs autonomously for 7+ days. P&L is positive after fees. No manual intervention required.
- **North star:** The system runs while you sleep and you wake up to see it made money. Not just once, but consistently.
